# 핵심 구성요소 (재배치안 v8 — 260702)

> **과제명(제안서 제목, 2026-07-02 확정)**: 「버추얼 성능개발(CAE) 통합 지능화·자동화 — AI가 해석하고, 3중 게이트로 검증한다」 ※ '플랫폼'은 제목에서 제외(본문 범위 선언·통합 절의 장기 비전으로만 사용). 제목 금지어: '하네스'(와이어링 하네스 오독 위험) · '이론값'(복잡형상 응력분포·동/정강성·피로수명은 닫힌 이론해 부재).

> **한 줄 요약**: 표준화된 해석 프로세스(전처리→해석→후처리)를 에이전트로 **엔드투엔드 자동화**하되, 사내 표준·경험을 주입한 **하네스**가 ①실행 레일이자 ②출력 검증 기준(ground truth)으로 작동하고 ③자가개선 루프로 지속 강건화되어 — **사람 개입을 최소화한 신뢰성 있는 자동화로 해석 업무의 리소스·시간을 구조적으로 절감**한다.

> **효과 사슬(이 인과를 본문 전체에서 유지)**: 하네스·강건화·3중 게이트(자동화 품질 장치) → **무개입 완주율↑ · 재작업↓**(자동화 품질) → **[1차 효과] 해석 맨아워·리드타임 절감** → [2차] 재작업·과도설계 감소(중량·원가) → [3차] 지식 자산화 · 프론트로딩 · 타 조직 확산

> **인터페이스 원칙(단일 대화 창구)**: 자동화 제어(2번) · 엔지니어링 추론 Q&A(3번) · 마일스톤 승인(2번 5) 운영 모드)이 **하나의 채팅 인터페이스**에서 이뤄진다 — 작업 공간과 대화 공간의 통합, 개발자의 IDE 코파일럿 패턴을 CAE로 이식한 **'CAE 코파일럿'**. 로컬 세션 컨텍스트 브리지가 이를 뒷받침(구현은 3번 4) 참조). ※ 제안서에는 사외 제품명 언급 없이 'IDE 코파일럿 패턴' 수준으로 일반화 표기.

> **왜 이 업무가 AI에 적합한가**: CAE는 해석 인풋덱·실행 로그·업무표준 문서·보고서 등 **텍스트 중심 업무**이며, 프로세스가 이미 표준화·문서화되어 있다. LLM 에이전트(생성·추론형 AI)가 다루기에 최적의 구조. ※ 예측형 ML이 아닌 생성·추론형 과제.

> **용어**: **하네스(Harness)** = AI 에이전트가 따라야 할 업무표준·절차·판단 기준·검증 장치의 총체(와이어링 하네스처럼 프로세스 전체를 관통·연결). **TDD 설계** = 각 단계의 통과 조건을 사전 정의하고, 만족해야 다음 단계로 진행. **무개입 완주율** = 반복 실행 단계에서 추가 개입 없이 전처리→해석→후처리 표준 절차를 완료한 작업 비율 — 자동화 품질의 핵심 지표(Phase 1에서 실측). **※ 사람의 최종 책임을 배제한다는 의미가 아니며, 공식 판정·설계 반영은 해석 엔지니어의 마일스톤 승인 후 진행.** **사례 카드** = 과거 해석 보고서에서 추출한 구조화 사례 단위(부품·하중조건·문제현상·원인·대책·정량 결과) — 유사 사례 검색·유추판단의 기본 단위.

## 1. 버추얼 성능개발 지식 하네스 — 에이전트 목표·검증 기준 주입 (RAG/wiki) 【기반: 신뢰의 원천】
**정체성: 자동화를 관통하는 '살아있는 하네스'. ①목표를 주입하는 실행 레일, ②출력을 검증하는 기준(ground truth), ③쓸수록 강해지는 자산. 존재 이유는 하나 — 2번 해석 자동화를 '믿고 맡길 수 있게' 만드는 것.**

- **목표 주입 (실행 레일)** — *[구현 완료]*
  - 사내 업무표준·성능개발 경험·실무 인풋모델을 에이전트에 주입해 명확한 목표와 절차 부여
  - 사내 보고서 문서 RAG / 해석 업무표준 가이드 RAG / 실무 해석 인풋모델 RAG
  - **데이터 선행조건 관점의 강점**: 업무표준·가이드가 이미 문서로 정비되어 있어 대규모 신규 데이터 구축 불필요 → RAG 인덱싱 수준으로 착수 가능
- **검증 기준 (ground truth)** — *하네스의 이중 활용*
  - 동일한 하네스 문서를 에이전트 출력의 검증 잣대로 활용 → 사내 표준 준수 여부 자동 대조 (2번 3) 게이트의 1층이 이 기준을 호출)
- **자가개선 강건화 루프** — *[PoC·설계 단계]*
  - 실행로그·에러 복구 경험·**엔지니어 승인/반려 이력**을 지식으로 자산화(LLM wiki)하여 하네스를 지속 강건화
  - **강건화 계측 지표**: 게이트 1회 통과율↑ · 동일 에러 재발률↓ · self-correction 반복 횟수↓ → **곧 2번 자동화의 무개입 완주율을 끌어올리는 동력**('쓸수록 절감 효과가 커진다'의 측정 근거)
- **기대효과(훅)**: 신규 인력 온보딩 기간 단축, 조직 지식 재사용률 향상, 암묵지의 형식지화(조직 자산화)

## 2. CAE 해석 프로세스 자동화 — 하네스 기반 엔드투엔드 실행 【몸통: 1차 효과의 발생지】
**정체성: 본 과제의 몸통. 해석 1건에 드는 반복 수작업(모델 생성·수정, 조건 설정, 실행, 결과 정리·보고)을 전처리→해석→후처리 전 구간에서 에이전트가 대체해 해석 리소스·시간을 직접 절감하는 층. 1·3번은 이 자동화를 고신뢰로 만드는 품질·지능 장치다.**

1) **Agent 기반 해석 프로세스 자동화**: 텍스트로 에이전트와 소통하며 CAE 프로세스 제어 *(H-Chat Pro 경유, 3번 Q&A와 동일한 단일 대화 인터페이스)*
   - 전처리: HyperMesh(해석 모델 생성·수정) — 텍스트 라이브 interaction, FE모델·설계정보·하중신호 분석
   - 해석: Abaqus/Nastran + nCode(피로해석) — MCP 및 내부 RPC로 해석 서버 활용
   - 후처리: 결과 시각화·분석, 보고서 작성(사내 표준 양식 = 하네스 문서 준수)
2) **하네스 준수 설계(TDD)**: 해석 프로세스는 표준화·문서화되어 있으므로, 창의적 이탈이 아니라 하네스(=1번)를 충실히 따르도록 TDD로 설계 → 실행 자체를 통제
3) **환각·과도설계 차단 3중 검증 게이트** — *저비용 사전 게이트 → 고비용 최종 관문 순 단계화*
   - **1층 (저비용·상시)**: 하네스 문서 대조 — 1번 ground truth 기준 표준 준수 자동 확인
   - **2층 (중간)**: LLM 추론 기반 자기검토·교차검증
   - **3층 (최종 관문)**: 물리 타당성·실적 검증 — 평형·단위·자릿수(오더)·분포 패턴 등 물리 법칙 기반 타당성 + 유사 부품의 검증된 실적 범위 대조 (+ 닫힌 이론해가 있는 단순 문제는 이론값 직접 대조 추가)
   - → **"환각 및 과도설계 차단!"** — 게이트 통과 = 재작업 없는 완주 → 무개입 완주율·리소스 절감에 직결 *(2·3층은 3번의 Adams 검증 AI-AE 2단계 방법론을 계승, 1층은 사내 표준 대조 층을 신설해 보강)*
4) **에러 자가복구 → 하네스 환류**: 실행 중 에러는 self-correction으로 복구(복구 성공 = 사람 호출 없이 완주), 그 경험은 1번 하네스로 환류되어 강건화(1번 자가개선 루프의 입력원)
5) **운영 모드 — 사용자군별 이원화 (Human-in-the-Loop 설계)**
   - **기본: 마일스톤 승인 모드** — 해석 엔지니어가 주요 마일스톤(모델 확정·해석조건 확정·결과 판정 등)에서 직접 검토·승인하며 진행(승인도 동일 채팅에서, **승인·반려 사유는 1번 하네스로 환류되어 자산화**)
   - **옵션: 자동 진행 모드(승인 bypass)** — 설계·시험 등 **해석 비전문가의 셀프서비스 사전검토**용. 자동 게이트(3항)는 항상 작동하며, 산출 결과는 '사전검토용'으로 구분하고 공식 판정 활용 시 해석 엔지니어 승인 경유. 비전문가도 동일 대화 인터페이스에서 질문·결과 확인 가능(3번 Q&A 층이 안전한 셀프서비스를 뒷받침)
   - → **게이트가 강건하기에 비전문가 모드가 가능**. CAE 역량이 해석팀 밖(설계·시험)으로 확산되는 **프론트로딩** 실현
- **기대효과(훅) [1차 효과]**: 전·후처리 맨아워 절감, 해석 리드타임 단축, 비전문가 셀프서비스로 해석 의뢰 대기 감소 — **산정 로직: 무개입 완주율 × 건당 절감시간 × 연간 해석 건수 = 연간 맨아워 절감**

## 3. AI-AE 엔지니어링 — 검증된 추론 방법론의 전이 *[검증 완료]* 【지능: 과제명의 '지능화' 축】
**정체성: 자동화 위에서 '엔지니어처럼 추론·판단'하는 지능 층. 자동화(2번)가 '절차'를 수행한다면, 지능화(3번)는 절차가 멈추는 지점의 '판단'을 수행한다 — 판단이 막히던 지점을 뚫어 무개입 완주율을 함께 끌어올린다.**

- **공통 실행 패턴 (Adams 검증 2단계 방법론의 일반화)**: 가설 생성(LLM 추론) → 검증 설계 → 솔버/이론 실행(2번 자동화 인프라 공유) → 대조 판정 → 결론·wiki 자산화(1번 환류). 추론 산출물도 2번 3) 3중 게이트(하네스 문서 대조·자기검토·솔버/물리 타당성 확인)를 통과 — **"판단도 검증된다."**
- **활용 국면 4가지 (해석 라이프사이클 기준)**
  1) **해석 전 — 해석 계획·하중 추론**: 설계변경·요구성능 → 해석 종류/하중케이스 선정, 하중 이력(.req/.rsp) 분석으로 주손상 방향·지배 하중케이스 도출(Damage Rose·rainflow) — *[PoC 근거 보유]*
  2) **해석 후 — 원인 진단(Why) 추론**: 응력집중·수명 미달의 원인 가설 수립(하중 경로·강성 분포·형상 불연속) → 가설별 간이 검증 디스패치 → 원인 확정
  3) **개선 — 대책안·가설-검증 추론**: 대책 후보(리브·두께·형상·재질) + 트레이드오프(중량·원가) 추론 → LLM 추론 기반 파라미터 스터디로 검증 케이스 최소화 — *[PoC 완]* → 4번 최적화로 연결
  4) **상시 — 모델·지식 Q&A**: 해석/차량 모델 심층 질의응답(모델링 기법·조건 유래), 과거 보고서 기반 유사 사례 유추 판단 — *[PoC 근거 보유]*
     - **구현 ① 세션 공유형 대화 인터페이스**: 로컬 세션 컨텍스트 브리지(MCP/내부 RPC)가 작업 중인 도구의 상태(모델 트리·선택 엔티티·하중케이스·최신 결과·실행 로그)를 조회 도구로 노출 → "이 부품", "방금 결과" 같은 지시어 질문에 재설명 없이 즉답. **원칙: 대형 FE 모델은 컨텍스트에 직접 담지 않고 질의 도구로 조회**
     - **구현 ② 사례 카드 기반 유추판단**: 과거 보고서에서 LLM이 사례 카드(부품·하중조건·문제현상·원인·대책·정량 결과)를 일괄 추출해 1번 wiki에 구조화 축적 → [유사 사례 검색 → 현재 케이스 대조 → 차이 보정 판단]의 3단 유추
     - **Q&A도 하네스 안에서**: 답변에 출처(하네스 문서·사례 카드) 인용 강제, 무출처 답변 차단(2번 3) 게이트 1층의 Q&A 적용) / 자주 묻는 질문 로그 = 지식 갭 신호 → 1번 자가개선 루프의 입력원(**"질문이 하네스를 키운다"**)
- **구현 도구 (반복 판단의 표준화)**: **Skill**(판단 절차의 문서화 — 예: '응력집중 원인진단' 체크리스트·판단 기준, 지식 하네스의 실행형) / **Command**(정형 분석 원클릭 — 예: /하중분석, /모델브리핑) / **Agent**(다단계 추론 오케스트레이션 — 가설 생성→검증 디스패치→판정)
- **Adams에서 검증한 AI-AE 2단계 방법론(LLM추론 + 솔버검증)을, 자동화 대상인 구조/피로(HyperMesh + Abaqus/Nastran + nCode)로 전이** — *검증된 방법론의 확장(전이 진행 중)*
- **기대효과(훅)**: 엔지니어링 판단 리드타임 단축, 판단 품질의 표준화(가설/근거/검증/결론 구조화), 예외 상황 자율 처리로 무개입 완주율 기여

## 4. 최적 설계안 제안 및 검증 — '적정 설계'를 만드는 개선 루프 【성과: 자동화·지능화가 만드는 결과물】
**정체성: 3번 추론과 2번 자동화가 결합해 '검증된 설계 개선안'을 산출하는 최종 성과 층. 게이트가 과도설계를 막고(신규 설계), 여유도 스캔이 기존 과잉 마진을 찾아내(기존 설계) — 넘치지도 모자라지도 않는 '적정 설계(right-sizing)'를 지향한다. ※ 제출 본문에서는 '확장 성과'로 압축 배치(상세는 프롬프트 지시 참조).**

- **설계 개선 루프 (공통 패턴)**: 자연어 문제 정의("피로수명 목표 미달, 중량 증가 최소로 개선") → LLM 정식화(설계변수·목적·제약 — **제약은 하네스 문서에서 자동 추출**: 최소 두께·재질·패키지·성능 기준 → '만들 수 없는 최적안' 원천 차단, **하네스의 3중 활용: 실행 레일·검증 기준·제약 소스**) → 엔진 선택·탐색 실행(2번 무인 자동화) → 후보 검증(3중 게이트) → **설계 언어 번역**(대책안·근거·트레이드오프) → 운영 모드 승인
- **3대 최적화 엔진** *(LLM 추론 탐색 + 수치 엔진 조합 — 예측형 ML 미사용)*
  - **치수(sizing)**: 3번 진단·사례 카드 기반 민감 변수 가설 → 유망 방향 우선 탐색으로 해석 케이스 감소 — *[PoC 완]*
  - **형상(shape)**: HyperMesh 모핑 도메인 정의, **3번 원인진단(응력집중·하중 경로)이 모핑 방향의 근거** → 국부 형상 개선(필렛·비드·단면) — *[계획 단계]*
  - **위상(topology)**: 자연어→SIMP 설정(설계영역·하중·체적분율), GPU 가속 실행 → 밀도 분포를 **리브 배치·살빼기 패턴으로 설계 번역**(STL 형상 후보 export) → 재해석 검증 — *[구현 완]* *(수치최적화, 딥러닝 아님)*
- **[신규] 과도설계 여유도 스캔 (상시 모드)**: 완료된 해석 결과에서 목표 대비 마진 과다 부위·부품을 스캔 → 경량화·원가절감 후보 자동 발굴 → 최적화 루프로 연결 — **"성능 미달을 고치는 것뿐 아니라, 과잉 성능을 찾아 깎는다"** *(문제가 없어도 가치를 만드는 모드, 규칙 기반 스캔 + LLM 판단)*
- **[신규] 설계 협의 패키지 자동 생성**: 산출물을 설계부문 협의용 자료로 자동 편집 — 대책 요약 / 근거(3번 진단+해석 결과) / 트레이드오프(성능·중량·변경비용) / 게이트 통과 증빙. **실무의 최종 산출물은 메쉬가 아니라 '협의 자료'** *(2번 보고서 자동화 재사용 — 본문에서는 2번 후처리의 연장으로 배치 가능)*
- **기대효과(훅)**: 최적화 소요 해석 케이스 수 감소율 *[PoC 근거]*, 설계 개선 리드타임 단축(진단→대책→검증 자동 루프), 경량화·원가절감 후보 발굴 건수(여유도 스캔), 설계 협의 준비 시간 절감, 과도설계 감소(중량·원가)

# 제안서 보강 섹션 (심사 기준 직결)

## 실현 기반 — 선행조건·보안·권한 *(실현가능성 근거)*
- **보안**: 전 과정 사내망 내 구동, 사내 승인 AI 서비스(**H-Chat Pro**) 경유 — 데이터 외부 반출 없음. 해석 서버는 내부 RPC/MCP로 연결, 하네스 문서(RAG)는 기존 문서 접근권한 체계 준수
- **데이터**: 핵심 입력(업무표준·가이드·인풋모델)이 이미 사내에 정비되어 있음 → 신규 대규모 데이터 구축 불필요
- **도구**: 사내 AI 도구(**H-Chat Pro**) 중심 구현 — **H-Chat Pro 경유 도구 실행 연동까지 PoC로 확인 완료**(막연한 계획이 아닌 실증). 확산 시에도 승인된 내부 API/RPC 범위 내에서 단계적 확대 → 공모전 '사내 AI 도구 활용' 요건의 실증적 충족
- **권한·책임**: 에이전트는 사용자의 기존 권한·솔버 라이선스 범위 내에서만 실행. 공식 해석 결과의 승인 권한은 해석 엔지니어에게 유지 — 비전문가 셀프서비스 산출물은 '사전검토용' 표시, 설계 반영 전 해석 엔지니어 검토 필수

## 단계별 로드맵 *(현재 → 확산)*
- **현재**: 구성요소별 소규모 파일럿 다수 완료(요소 단위 PoC 검증) — 지식 RAG, Agent 자동화 요소, AI-AE 방법론(Adams), 하중 이력 분석, 세션 브리지형 어시스턴트, **H-Chat Pro 도구 실행 연동**, 최적화(치수·위상) 등 ※ 세부 파일럿 목록은 근거 세션에서 구체화
- **Phase 1**: **구조/피로 대표 프로세스 2종을 우선순위에 따라 순차 적용** — ①부품 구조강도 해석(단순 프로세스 우선 검증) → ②샤시 부품 피로해석(= 하중이력 기반 nCode 피로해석, 하중신호 분석을 포함한 풀 파이프라인). 각 프로세스에서 하네스→실행→게이트→환류 루프 전체를 엔드투엔드 검증하고, **무개입 완주율·건당 절감시간을 실측해 1차 효과 정량화**
- **Phase 2**: 구조/피로 업무 전반 확장 + 비전문가 셀프서비스 모드 오픈(설계·시험 부문)
- **Phase 3**: 타 CAE 도메인(NVH·충돌·열유동 등) 및 표준화된 문서 기반 업무 전반으로 하네스 방법론 이식

## 일하는 방식의 변화 *(창의혁신성 근거)*
- **해석 엔지니어**: 반복 실행자 → **목표 정의자·검증 승인자** (마일스톤 승인 모드)
- **설계·시험 엔지니어**: 해석 의뢰 후 대기 → **셀프서비스 사전검토** (자동 진행 모드) — CAE 프론트로딩
- **조직**: 개인의 암묵지 → 하네스라는 **형식지 자산**으로 축적, 쓸수록 강해지는 시스템
