## §0. 핵심 요약 (Executive Summary)

**한 문장.** 전략 원본 3종이 내세운 "**상용 전처리기(HyperMesh)를 라이브로 직접제어 + 이론값 가드레일을 동시 달성한 유일 시스템**"이라는 서사는 외부 선례와 코드 실측 앞에서 **상당 부분 과장으로 판명**되었으나, 그 **방향성(생성자와 독립된 검증 레이어 + 자동차 내구 도메인)**은 여전히 유효하며, 이를 **정직하게 좁혀** 실증하는 것이 현 시점의 올바른 기술개발 계획이다.

**무엇이 무너졌나 (과장):**
1. **"RPC 라이브 직접제어가 해자" → 반증.** 〔외부검증〕 동일 HyperMesh를 표준 MCP+GUI 소켓(포트 47881)으로 라이브 제어하는 `times1234/hypermesh-mcp`가 실재하고, `Abaqus-Control-MCP`(136★)·`COMSOL MCP`(MPh, 482★)·**Ansys 공식 `PyMAPDL-MCP`(gRPC 라이브세션 + MCP 어댑터)**가 모두 라이브 세션 제어를 한다. Altair 직원이 LangChain+OpenAI로 동급 HyperMesh 에이전트를 며칠 데모로 만든 사례까지 있어 **복제 난이도도 해자가 못 된다**. 더구나 MCP의 데이터계층 자체가 JSON-RPC 2.0이라 "RPC vs MCP" 대립 프레임이 부정확하다.
2. **"이론값 가드레일이 결과 날조를 사전 차단하는 1급 가드레일" → 부분적 과장.** 〔실측〕 정작 본체인 hm-assistant는 **동어반복(solver=validator)**이 미해결이다 — `agent/cae_planner.py:10`이 닫힌형 `mock_hm_server._evaluate_response_model`을 그대로 "솔버"로 import해 검증하므로 구조적으로 무조건 PASS다. 독립 검증이 실제로 작동하는 것은 **별도 시스템 agent_cae3**뿐이고, 그조차 검증기와 솔버가 **같은 LLM 추출 파라미터를 공유**해 입력오류는 못 잡는다(epistemic 비독립). 〔외부검증〕 FEABench는 "실행 가능 88% vs 엄격 수치정답 실질 0"으로 **이론값 검증이 미해결 프런티어**임을 입증한다.
3. **"E2E 자율·정량 효과가 우리 차별점" → 반증.** 〔타팀〕 공모전 예시1이 이미 **실 자동차 부품(JG 휠)**에서 ANSA 소켓 제어→메시→Input Deck→배치솔브→ODB→HTML까지 E2E를 자기보고 실증했고, 보고서가 인용하려던 정량수치(대기 0분·Nastran 대비 0.001%·후처리 수시간→수분)는 **전부 예시1의 자기보고**다. 이를 자사 가치로 옮기면 같은 심사장에서 즉시 반증된다(층위 혼동).

**무엇이 살아남나 (방어 가능한 좁은 핵심):**
- **검증 레이어로서의 차별.** 〔외부검증〕 상용 벤더(Altair CoPilot=RAG Q&A 등)와 선례 5종 어느 것도 "해석 결과를 **생성자와 분리된 물리 이론식 oracle**로 채점"하지 않는다. 단 이는 **선형·정준 형상(빔·판·브라켓)에 한정**되며 "만능 검증기"가 아니다.
- **4중 교집합.** 〔비평〕 'HyperWorks 생태계 × 라이브 세션 × closed-form 이론값 게이트 × 자동차 부품'은 자체 카탈로그 130건 중 0건(단, 부분적 음성근거).
- **nCode 내구피로라는 미답 영역.** 〔외부검증〕 LLM-CAE 종합 서베이(arXiv 2509.11447)가 fatigue/S-N/rainflow를 **0회 언급** — 사용자가 nCode DesignLife를 보유한 점과 결합하면 가장 자연스러운 본진. 단 **현 기능이 아니라 로드맵**이며 피로는 closed-form 이론값이 모호해 검증방식이 이질적이다.

**현 구현 실측 요약.** 〔실측〕 hm-assistant = Phase-1 PoC(45 tools, 142 PASS+1 SKIP, Mock end-to-end 3사례) + Phase-2 진행(RPC 플러밍만 사내 HyperMesh 2026.1 스모크 통과; bridge 도메인 명령·사내 LLM 어댑터는 `NotImplementedError` stub). **실솔버 실행 0건, 런타임 LLM은 사외 Gemini → 사내도구 규정 컴플라이언스 갭.** agent_cae3 = REAL Abaqus(`abaqus cae noGUI`→.odb) + 독립 closed-form `theory_validator`(actual을 외부입력으로 채점) + ~610 테스트. **두 시스템이 함께 돌아간 적은 없음(통합리스크 #0).**

**핵심 권고(3줄).**
1. **포지셔닝을 "유일/만능"에서 "정직하게 좁힌 검증 레이어 + 자동차 내구 도메인"으로 재정의**하고, 모든 광범위 주장에 충돌 선례를 병기한다(부록 E).
2. **현재 시점 유효 아키텍처 = 내부 전용 세션엔진 + 외부 MCP 어댑터 + 교체가능 LLM(사내 정합) + 도메인 typed tools + 다층 검증게이트** — 이것은 뒤늦은 추종이 아니라 **Ansys 공식 스택이 이미 검증한 표준 패턴**이며 원본 로드맵·비평이 수렴한 결론이다.
3. **6월말→9월말 실증의 1순위 = 동어반복 해소(실솔버 1-run) + 정량 KPI 측정(성공률·개입횟수·오결과 차단율·게이트 FAIL 검출률) + 사내 LLM 어댑터 실구현.** 이 셋이 없으면 모든 가치 주장이 미입증으로 남는다.

### 2.1 "RPC 라이브 직접제어가 독보적·복제곤란 해자다" → **refuted (확신: 높음)**

〔외부검증〕 주장을 4요소로 분해하면 3/4가 직접 반례로 무너진다.

- **'유일/독보' — 반증.** `times1234/hypermesh-mcp`는 **동일 소프트웨어 HyperMesh**를 표준 MCP + GUI 소켓(포트 47881)으로 **가시 라이브 세션에 Tcl을 메인스레드 주입**한다(대상 SW가 같아 가장 강한 반례). `Whfkl/Abaqus-Control-MCP`(127.0.0.1:48152, 136★)는 실행 중 Abaqus/CAE 세션의 `mdb`·`session`·`odb`에 직접 접근한다.
- **'복제 곤란' — 반증.** Altair 직원이 LangChain+OpenAI로 동급 HyperMesh 에이전트를 **개인 데모로 며칠 만에** 구현했다. 메커니즘은 상품화·복제 가능.
- **'해자=RPC' — 반증.** 〔외부검증〕 MCP 데이터계층 = JSON-RPC 2.0(공식문서). **Ansys 공식 `PyMAPDL-MCP`는 "라이브 gRPC 세션엔진 + MCP 어댑터" 하이브리드를 벤더가 직접 구현** — "RPC vs MCP"는 택일이 아니라 **층위 차이**이고, RPC 자체는 차별점이 못 된다.
- **'~5% 희소영역 선점' — 약화.** 5%는 내부 카탈로그 추정치이며, 선점도 times1234·벤더 데모로 흔들린다.

**살아남는 좁은 진실** 〔외부검증〕: 확인된 LLM-CAE 학술 12편은 모두 배치/headless/입력파일 생성이고, "**지속적 라이브 인터랙티브 전처리기 GUI 세션을 직접제어 + 물리 이론값 게이트**"의 결합은 학술에서 미확인이다. 그러나 이 희소성은 오픈소스(times1234)로 이미 깨졌으므로 **'독보적 해자'를 구제하지 못한다**.

> **정직한 재표현(이후 본문·발표 공통):** "TCP+JSON-RPC로 HyperMesh 라이브 세션을 제어한다"는 **사실 진술은 유지**하되 '독보적·유일·복제곤란 해자' 표현은 삭제한다. 충돌 선례(times1234·Abaqus-Control-MCP·PyMAPDL-MCP)를 반드시 병기하고, "RPC vs MCP 대립" 프레임을 버린다.

### 2.2 "이론값 가드레일이 결과 날조를 사전 차단하는 1급 가드레일이다" → **partially-true (확신: 높음)**

강한 수식어 4개("차별적·지속가능한·해자·1급 사전차단")는 과장이나, **문제유형을 한정한 좁은 핵심**은 방어 가능하므로 refuted가 아니다.

- **작동성 — 본체는 동어반복** 〔실측〕: hm-assistant의 현 수렴 루프는 `agent/cae_planner.py:10`이 닫힌형 `mock_hm_server._evaluate_response_model`을 그대로 "솔버"로 import해 `abs(u_y − target) ≤ tol`을 검사한다. **솔버=검증 출처가 동일** → 구조적 0% 오차·무조건 PASS. 게다가 LLM 없는 mock 스켈레톤이라 라이브 에이전트가 호출하지도 않는다 → **'실증된 가드레일'이 아니라 '설계된 가드레일'**(실솔버 검증 0건).
- **비유일성** 〔원본 자인〕〔외부검증〕: 자사 Blueprint 8쪽 매트릭스조차 VFEAgent를 같은 가드레일 축 'full'로 표기한다. 학술 선례 다수(FEABench의 Analytical–Numerical Consistency, VFEAgent L1~L4) → "세계최초/유일" 불가.
- **사전차단·1급 — epistemic 비독립** 〔실측〕: 가장 성숙한 agent_cae3조차 `theory_validator`가 actual(.odb 실측)을 외부 인자로 받아 독립 closed-form으로 채점하는 **code-vs-code 독립**(좌표계·단위·BC 버그는 검출)은 실재하나, theory와 actual이 **같은 LLM 추출 파라미터(length/width/height/load/E)를 공유**하므로 입력오류가 양쪽에 전파되면 통과한다 → **'날조 원천 차단'이 아님**.
- **한정성 — 만능 검증기 불가** 〔외부검증〕: 선형 closed-form(빔·판·브라켓)에만 유효. 비선형·접촉·충돌·열-구조 연성은 이론값이 모호. agent_cae3 코드 주석도 *"engine-mount/stabilizer/bracket 형상은 빔 이론식에 대입하지 말 것"*이라 명시하고, 허용폭 0.5x~1.5x라 **거친 오류(단위·BC·중대 모델링오류)만 차단**한다. FEABench의 strict 정답 실질 0(1/15)이 "미해결 프런티어"를 입증.
- **검증과학** 〔외부검증〕: 검증자가 생성자와 독립이 아니면 검증은 무력하다 — Huang et al.(ICLR'24)은 외부 oracle 제거 시 자기수정 효과가 소멸함을 보였다(GPT-4 GSM8K 95.5→89.0%). "실행됨 ≠ 물리적 정답"(FEABench), 도구사용 환각(ToolBeHonest GPT-4o 37/100). **RPC '성공' 자기보고를 신뢰 근거로 쓸 수 없다**.

> **정직한 재표현:** 이론값 검증을 '유일·만능 해자'가 아니라 **문제유형별 검증 레이어**로 서술한다 — *"선형·closed-form 영역에서 솔버와 독립된 이론식 oracle로 좌표계·단위·BC 오류를 차단하는 코드 게이트를 두고, 비선형·접촉·피로는 실 독립솔버·시험데이터로 닫는다."* 방어선 4개를 함께: (a) 비유일성(선례 병기), (b) 한정성(선형만, 나머지는 caveat 레이어), (c) 구현상태 정직표기(hm-assistant는 동어반복 mock = '설계' 단계, 독립검증은 agent_cae3에만), (d) 효과 KPI 공백(성공률·차단율 측정 전까지 '1급'이라 부르지 않음).

### 2.3 "사내 경쟁자·외부 선례 대비 차별적이다" → **partially-true / weakened (확신: 중간)**

좁은 의미(특정 조합 존재)로만 참이고, 원본의 강한 차별화 서사는 대부분 무너진다. **가장 강한 반증은 경쟁자 원문(예시1)**이다.

- 〔타팀〕 **예시1**(예시케이스1.txt 직접 판독)이 Hyper-Pilot의 핵심 차별축을 거의 전부 보유: ① **ANSA를 IAP '소켓 통신'으로 직접제어**(=상용 전처리기 라이브 소켓 제어, 바로 그 축), ② 메시→INP→Abaqus/Nastran MCP 배치→ODB→HTML **E2E 자율**, ③ **Nastran 교차솔버 검증**(K_dyn 오차 0.001~0.00% 주장), ④ `Lesson.learned.md` 자가자산화. 게다가 실 자동차 부품(JG 휠 동강성)에서 정량결과까지 자기보고.
- **성숙도 역전** 〔실측〕: 예시1은 실부품 전 파이프라인 "검증 완료"+정량결과를 표방하나, Hyper-Pilot은 Mock-only PoC(실솔버 0건·KPI 전무).
- **컴플라이언스 역전** 〔실측〕〔외부검증〕: 경쟁자 2종은 사내 허용 **Claude Code 기반**. Hyper-Pilot 런타임 LLM 기본값은 **사외 Gemini**(internal_adapter.py = stub) → 공모전 '사내 도구 only' 규정상 오히려 결격 리스크.
- 〔타팀〕 단, **예시2**의 검증은 *"결과 파일 정상 산출 여부 1차 검증"*(executability-only) — FEABench가 반증한 바로 그 함정 — 이라 **이론값 게이트는 예시2 대비 실질 차별점**이다.

**방어 가능한 좁은 차별 2가지로 한정** 〔비평〕: (a) 정적 구조에서 **생성자와 분리된 closed-form 이론값 oracle로 실솔버(.odb)를 독립 채점**(예시1의 교차솔버는 입력추출 오류를 양쪽에 공유; closed-form은 두 솔버가 공유할 오류와도 독립까지만), (b) HyperMesh/OptiStruct·**nCode 피로**라는 미답 차선. 단 (a)는 코드-대-코드 독립이며 미실증, (b)는 현 기능이 아니라 로드맵.

> **정직한 재표현:** "유일하게 차별적"이 아니라 **"좁고 특정한 조합에서만 방어 가능한 차별점"**. 예시1 정면 비교를 회피하지 않고 명시하는 것 자체를 신뢰자산으로 전환한다.

### 2.4 "보고서가 제시할 정량 가치는 방어 가능하다" → **refuted (확신: 높음)**

**치명적 발견** 〔타팀〕: 흔히 인용되는 4개 수치 — "대기 0분 / Nastran 대비 0.001% / 후처리 수시간→수분 / 외주비 절감" — 은 **전부 경쟁자 예시1의 자기보고**(예시케이스1.txt line 60·78·79·80·85)다. 자사 가치로 옮기면 **층위 혼동**(프로젝트 CLAUDE.md 금지)이며 같은 심사장에서 즉시 노출된다.

각 수치를 그 자체로 봐도 방어 불가:
- **"Nastran 대비 0.001%"**는 AI 정확도가 아니라 **동일 입력덱에 대한 솔버 간(Abaqus↔Nastran) 교차일치**다. 선형 모달에서 상용 솔버끼리 일치하는 건 당연하며, **BC를 틀리면 두 솔버가 똑같이 틀린 값에 0.001%로 '일치'**한다(기본값 함정). 원문도 0.00%(line60)와 0.001%(line79)로 불일치하고 n=1 선형 단일케이스다.
- **"후처리 수시간→수분"** 〔외부검증〕: 하위작업 70~90%↓류는 대부분 **벤더 마케팅(독립검증 없음)**. ODB→csv→html 스크립트화 자체는 그럴듯하나 자사 실측이 아니다.
- 〔실측〕 Hyper-Pilot은 어떤 수치도 자사 증거로 뒷받침 못 한다(Mock-only, KPI 전무, 동어반복).

**그러나 방향성(절감 자체)은 외부 프록시로 약하게 지지된다** 〔외부검증〕〔추정 단서 필수〕:

| 프록시 지표 | 값 | 출처/단서 |
|---|---|---|
| 시뮬레이션 주도설계: 개발기간 단축 | ~29%↓ | Aberdeen 2017(자가보고, Best-in-Class) |
| 〃 물리 시제품 감소 | ~27%↓ | 〃 |
| 〃 ECO(설계변경) 감소 | ~21%↓ | 〃 |
| 문서작성: 시간/품질 | 시간 ~40%↓ / 품질 ~18%↑ | Noy-Zhang, *Science* 2023 RCT (일반 사무문서 → 엔지니어링 보고서는 별도 실증 필요) |
| 해석 이양 시 납기달성 | +17%p | Aberdeen(병목 완화·민주화 방향) |
| AI 코딩 어시스턴트 생산성 | **−19% ~ +55%** | GitHub Copilot/Google/METR RCT — **맥락 의존 극심, 음(−)효과 공존** |

> **정직한 재표현:** 정량 가치는 단일 헤드라인이 아니라 **출처·표본·측정방식을 병기한 시나리오 밴드**로 — 보수(한 자릿수~20%) / 중간(20~40%) / 낙관(50%+, 좁은 하위작업·그린필드 한정). 예시1 수치는 차용하지 않고 "타팀 벤치마크"로만 귀속한다. 후처리는 "반복 후처리 자동화로 단축"까지만 말하고, 자사 KPI(단계별 before/after 소요시간·성공률·개입횟수·오결과 차단율·게이트 FAIL 검출률)를 **통제비교 실측**으로 약속한다.

## §3. CAE × LLM 업무 프로세스 자동화 — 현재 지형 (검증 기반)

### 3.1 자연어–CAE 연동 방식 5분류, 외부 현실로 재검토

원본의 5분류 자체는 유용한 지도다. 단 비중(%)은 내부 추정이고, "소켓/RPC가 희소=해자"라는 함의는 §2.1에서 무너졌다. 현실을 반영해 재정리하면:

| 연동 방식 | 〔원본〕 비중 | 실제 성격 〔외부검증〕 | 한계 |
|---|---|---|---|
| 코드/스크립트 생성 | ~45% | 구현 쉬움, 솔버 무수정 | 환각·디버깅·실행컨텍스트 분리. "실행됨≠정답"(FEABench) |
| **MCP 도구 호출** | ~20% | **표준 도구계층(데이터계층=JSON-RPC 2.0)**, 클라이언트 확장성. 벤더·오픈소스 급속 채택 | 프로토콜이 검증·안전을 보장하지 않음. `execute_script` 하나만 노출하면 "코드생성에 MCP 포장지" |
| 직접 API 호출 | ~15% | 검증된 함수만 호출, 안정 | 라이브러리 구축비용↑, 유연성↓ |
| 임베디드 어시스턴트 | ~10% | 벤더 Copilot/챗봇, UX 우수 | 대개 **RAG Q&A**(Altair CoPilot 확인) — 자율 실행 아님 |
| 소켓/RPC 직접제어 | ~5% | 깊은 세션 제어, 낮은 추상화 손실 | **희소성 해자 아님**(times1234·PyMAPDL-MCP gRPC 등 라이브 제어 다수) |

**핵심 통찰** 〔외부검증〕: "내부 = 전용 세션 브릿지(RPC/소켓), 외부 = MCP 어댑터"는 **권고가 아니라 이미 검증된 표준 패턴**이다(Ansys 공식 PyMAPDL-MCP가 gRPC 라이브세션 + MCP를 결합). RPC와 MCP는 택일이 아니라 **층위가 다른 결합 대상**이다.

### 3.2 CAE 워크플로 단계별 자동화 성숙도

| 워크플로 단계 | 자동화 성숙도(업계 일반) | 검증(정답 판정) 난이도 | 비고 |
|---|---|---|---|
| CAD 정리·형상 단순화 | 중 | — | 하이브리드 라우팅(단순=생성/복잡=임포트)이 실무 관건 |
| **메싱** | 중~상 | 중(메시 품질지표 존재) | times1234가 기어인식·risk 배칭·롤백 등 **메싱 의사결정 깊이에서 앞섬** → 메싱을 헤드라인으로 내세우면 외부 선례에 노출 |
| 물성·BC·하중 | 중 | **상** | 과거 해석자산(검증된 BC/하중/물성)의 자산화가 핵심(예시1·예시2 공통 서사) |
| 솔버 실행(배치/잡) | 상 | 하(실행 성공 여부) | "실행됨"은 쉬움 |

| **보고서 생성** | 중~상 | 중 | 단 "범용 보고서 자동화"는 금지주장(특정 부품유형 한정으로 좁힘) |
| DoE·최적화 루프 | 중 | 중~상 | 24/7 광역 탐색 = 인간이 안 하던 새 가치(§4.3) |
| **물리 검증(이론값 정합)** | **하(미해결 프런티어)** | **최상** | FEABench strict 정답 실질 0 — 여기가 진짜 부가가치이자 가장 어려운 곳 |

요지: "실행 자동화"는 상향 평준화되었고 **차별은 가장 오른쪽 열(물리 검증)**에서 난다. 그런데 그 열이 가장 어렵고, 본체(hm-assistant)에서는 아직 동어반복 mock 단계다.

### 3.3 신뢰성·검증의 현주소 — 업계가 못 푼 프런티어 〔외부검증〕

- **FEABench**(arXiv 2504.06260, Google): API 호출 실행가능 **88%**에도 **엄격 수치정답은 실질 0(1/15)**. "실행됨 ≠ 물리적 정답"을 정량 입증. LLM-CAE를 "프런티어를 밀어야 할 미해결 과제"로 규정.
- **VFEAgent**(arXiv 2605.28978): ReAct 멀티에이전트 + 계층검증(L1~L4, 강체구속/범위 추정). 실행성·신뢰성을 **상대·정성 평가**로 자기보고(시간·ROI 정량치 없음). FEABench와 성공기준이 달라 **둘을 함께 인용**해야 분야 실상이 보인다.
- **자기검증의 함정**(검증과학): 검증자가 생성자와 독립이 아니면 무력(Huang et al., ICLR'24 — 외부 oracle 제거 시 GPT-4 GSM8K 95.5→89.0%). LLM-as-judge 자기선호 편향, 도구사용 환각(ToolBeHonest). → **동어반복(solver=validator)은 일반 원리상 무력한 검증**이며, 이를 깨려면 입력을 공유하지 않는 **독립 oracle**이 필요하다.
- **MCP 보안**: 로컬 바인딩/Origin 검증/DNS rebinding 경고(공식 transport 문서), tool descriptor 공격·품질 문제(arXiv 2512.06556·2602.14878 〔추정·미1차확인〕). 자율 실행은 승인·dry-run·allowlist·rollback·audit를 **구현자가 설계**해야 한다.

### 3.5 상용 벤더 동향과 fast-follow 리스크 〔외부검증〕

- **Altair CoPilot** = "Altair 온라인 도움말·지식소스 기반 질의응답"(공식 도움말 확인) — **RAG Q&A**이며 메시/솔버 실행 언급 없음. → 현재 벤더는 "검증축"을 비워두고 있다(자사 기회).
- **그러나 추월 위협 실재**: Altair 직원의 며칠 데모(복제 용이), **Siemens의 Altair 인수 완료**(HyperMesh가 Simcenter 계열로) → 벤더가 마음먹으면 fast-follow. Dassault Virtual Companions, Ansys(Synopsys 산하) 등도 동일 방향. **단일 벤더(HyperMesh) 종속**은 기술 해자인 동시에 운영·전략 리스크다(§5.7).
- 〔미확인〕 HyperWorks 2026의 자율 에이전트 탑재 여부, Ansys 공식 MCP 블로그 본문 — 1차확인 실패. 확정 서술 금지.

## §4. CAE × LLM이 만드는 가치 예측 — 효율 향상과 새로운 가치

> 원칙: "기존 업무를 빠르게(효율)"와 "전에 없던 가치(질적 변화)"를 **분리**한다. 모든 수치는 시나리오 밴드 + 출처 단서로만 제시한다(§2.4).

### 4.1 직군별 업무 프로세스 변화 (산업 일반 패턴)

CAE 병목은 "해석 전문성의 소수 집중"이며, 자연어 에이전트는 그 병목을 직군별로 다르게 푼다. (아래 직군 프레임은 〔타팀〕예시1이 잘 articulate했으며, 산업 일반에서도 관찰되는 구조다.)

| 직군 | 현재(병목) | 자동화 후(기대) | 전제 조건 |
|---|---|---|---|
| **설계 엔지니어** | 설계안→해석팀 의뢰→스케줄 대기→결과. 수정→재의뢰 반복으로 리드타임 팽창 | 설계 직후 **자체 1차 해석**으로 성능 즉시 확인 | 결과 **신뢰성 검증**이 없으면 위험(잘못된 해석을 믿는 비용) |
| **시험 엔지니어** | 시험–해석 정합 검증이 해석팀 협업에 의존 | 모델 확보 시 **자체 시험–해석 상관** 검증 | 시험데이터 연동·검증 레이어 |
| **해석 엔지니어** | 반복 셋업·후처리·보고서가 시간 대부분 | 단순 업무 자동화 → **고도화·선행연구 집중** | 자산화된 노하우(BC/하중/물성)와 게이트 신뢰 |

**핵심 단서**: 이 가치는 모두 "**검증이 신뢰할 만하다**"를 전제로 한다. 검증 없는 민주화는 "잘못된 해석의 대량 생산"이 되어 오히려 위험하다 — 그래서 §2.2의 검증 레이어가 가치의 **전제이자 차별점**이다.

### 4.2 정량 효율 예측 — 시나리오 밴드 (출처 병기)

자사 실측 KPI가 없으므로(공통 약점), 외부 프록시로 **방향성만** 밴드화한다. 자사 수치는 9월 실증 후 채운다.

| 가치 영역 | 보수 | 중간 | 낙관(조건부) | 프록시 근거 〔외부검증/추정〕 |
|---|---|---|---|---|
| 반복 후처리·보고서 초안 | 10~20%↓ | 30~50%↓ | 70%+↓(정형 단일유형) | Noy-Zhang 문서 40%↓; 후처리 스크립트화 |
| 해석 리드타임(대기 포함) | 한 자릿수% | 20~30%↓ | 의뢰-대기 구간 대폭(그린필드) | Aberdeen 개발기간 ~29%↓·납기 +17%p |
| 설계변경(ECO) 재작업 | — | ~20%↓ | — | Aberdeen ECO ~21%↓ |
| 엔지니어 코딩·스크립팅 | **−19%~+55%(불확실)** | — | — | AI코딩 RCT(맥락의존 극심) |

> ⚠️ 모든 행에 단서: Aberdeen=자가보고, Noy-Zhang=일반 사무문서(엔지니어링 보고서 별도 실증), AI코딩=음효과 공존. **CAE×LLM 자율에이전트의 직접·독립 ROI 벤치마크는 사실상 부재**(자체 카탈로그도 빈틈 #14로 지목). 따라서 효율은 "입증된 사실"이 아니라 **정량 PoC로 검증할 가설**이다.

### 4.3 새로운 가치 — "빠른 옛일"과 구분되는 질적 변화

| 새로운 가치 | 왜 '새로운가' | 단서·경계 |
|---|---|---|
| **검증 레이어 자체** | 결과를 물리 oracle로 채점해 **오결과를 차단**하는 신뢰 계층 — 기존 자동화엔 없던 층 | 선형 한정·KPI 입증 전제. 이것이 가장 방어 가능한 새 가치 |
| **해석 민주화** | 비해석가(설계·시험)가 1차 해석을 수행 | 검증 신뢰가 전제(없으면 위험) |
| **지식 자산화** | 개인 노하우(BC·하중·물성)→조직 공유 자산. 인수인계·신입교육 비용↓ | 〔타팀〕예시1·2도 동일 주장 → 차별점 아님, '기본기' |
| **24/7 자율 DoE/탐색** | 인간이 안 하던 광역 설계공간 탐색 | 솔버 비용·검증 신뢰가 한계 |
| **추적성·감사로그** | 자율 실행의 승인·rollback·audit이 규제(ISO 26262/NCAP) 대응 자산 | 설계 필요(프로토콜이 보장 안 함) |

**예측형 AI 경계선(중요)**: "실시간 예측·디지털트윈·서러게이트"는 **학습된 surrogate/neural operator/ROM** 으로 구현되며, 이는 **공모전 제외 대상(예측형 AI=ML/DL)**이자 본 보고서 범위 밖이다. CAE×LLM의 가치는 **LLM 오케스트레이션 + 솔버 코드/수학 생성 + 이론값 검증**에서 나오며, 예측코어를 ML로 대체하는 순간 규정 위반·성격 변질이 된다. 이 경계를 명시적으로 긋는 것이 AI과제적합성의 핵심이다(§5.6).

### 4.4 가치 실현의 전제·리스크

1. **검증 신뢰성 입증** — 가치의 전제. KPI(성공률·개입횟수·**오결과 차단율**·게이트 FAIL 검출률) 없이는 민주화가 위험.
2. **데이터·보안(IP)** — CAD/설계데이터가 LLM에 노출되는 리스크 → **사내 LLM(H-Chat Pro) 필수**, 사외 반출 불가.
3. **단일 벤더 종속** — HyperMesh API 변경·라이선스·버전 호환·세션 안정성이 제품 안정성 직결.
4. **조직 수용** — "AI 해석을 믿어도 되는가"의 신뢰 구축(추적성·인간 승인 단계).

## §5. 현재 시점에서 유효한 기술개발 계획 (핵심 로드맵)

> 전제: 제안 아키텍처(내부 RPC + MCP 어댑터 + typed tools + 다층 검증게이트 + nCode)는 **각 요소가 선례로 실현 가능**하나, "동어반복 해소 완료"·"현재 유효"는 과장이다(§2.6 #5). 따라서 **"선례로 뒷받침되는 실현 가능한 계획"**으로 정직하게 서술하고, 실증으로 닫는다.

### 5.1 설계 원칙

1. **정직성 우선** — 모든 주장에 충돌 선례·방어선 병기(부록 E). "유일/만능/세계최초" 금지.
2. **검증 가능성 우선** — 기능보다 **오결과 차단율** 같은 KPI를 먼저 만든다. 동어반복(solver=validator) 구조를 코드에서 제거한다.
3. **하이브리드 표준 채택** — 내부 전용 세션엔진 + 외부 MCP 어댑터(= Ansys 공식 패턴). "뒤늦은 추종"이 아니라 **선례·원본 로드맵·비평이 수렴한 합의**임을 근거로 제시.
4. **문제유형별 한정** — 검증을 "만능"이 아니라 레이어로 분리(선형 closed-form / 비선형·접촉은 caveat·실솔버·시험).
5. **사내도구 정합** — 런타임 LLM은 사내(H-Chat Pro), 에이전트 표면은 가능하면 Claude Code(허용 도구).

### 5.2 타깃 아키텍처

```
[자연어 지시]
   │  (사내 LLM: H-Chat Pro 어댑터 / 에이전트 표면: Claude Code 권장)
   ▼
[오케스트레이션]  ── ReAct/계획·도구호출 (LangGraph는 '라이브러리'; AI도구 규정 대상 아님 → §5.6 확인)
   │
   ├─▶ [외부 MCP 어댑터]  ── 표준 도구계층(생태계 호환·채택성). typed tools만 노출
   │        예) import_cad · cleanup_geometry · generate_mesh · check_mesh_quality ·
   │            assign_material · define_loadcase · run_solver · validate_result · export_report
   │        ✗ execute_script 하나만 노출 금지(= 코드생성에 포장지)
   ▼
[내부 전용 세션엔진]  ── HyperMesh hwx (TCP+JSON-RPC). 깊은 세션상태·잡관리·오류복구·rollback
   ▼
[솔버 실행]  ── OptiStruct/Abaqus (실솔버)
   ▼
[다층 검증게이트]  ◀── 동어반복 회피의 핵심
   L0 실행성(성공 여부)
   L1 안정성·정정성(단위·좌표계·강체운동)        ← VFEAgent식 사전검사, 비용 0
   L2 closed-form 이론값(선형 빔·판·브라켓)       ← agent_cae3식 독립 oracle(actual 외부입력)
   L3 실솔버 교차·메시수렴(2nd solver/수렴차수)
   L4 시험데이터 상관 · nCode 피로(장기)
   └ 문제유형 라우팅: 선형→L2 PASS / 비선형·복잡형상→REFERENCE_ONLY·caveat
   └ inject_error: 'PASS 재현'이 아니라 'FAIL 검출 능력'을 증명(동어반복 반증)
   ▼
[보고서·추적성]  ── 부품유형 한정 보고서 + 승인·audit·rollback 로그
```

핵심 변경점(현 코드 대비): ① `internal_adapter.py` stub → **사내 LLM 실구현**(컴플라이언스), ② bridge 도메인 명령 stub → **실 hwx 연결**, ③ `cae_planner`의 mock import 제거 → **독립 검증기 배선**(동어반복 해소), ④ MCP 어댑터 신설(typed tools).

### 5.3 진짜 해자의 재정의 (입증 대상 포지셔닝)

"RPC가 해자"가 아니라, **다음 교집합의 결합 품질**이 해자다 — 단, 아직 **'입증된 해자'가 아니라 '입증 대상 포지셔닝'**이다:

1. **생성자와 분리된 다층 검증 레이어**(L1~L4) — 선례(times1234 포함)는 검증이 메시·기하 품질에 그쳐 **물리 게이트가 부재**.
2. **4중 교집합** — HyperWorks 생태계 × 라이브 세션 × closed-form 게이트 × 자동차 부품(자체 카탈로그 130건 중 0건; **부분적 음성근거** 단서).
3. **nCode 내구피로** — 서베이 0건 언급 × 사용자 nCode 보유 = 미답 화이트스페이스(현 기능 아님, 로드맵).
4. **사내도구·한국어·현장 워크플로** — 글로벌 벤더 범용 Copilot이 세밀히 대응 못 하는 로컬 밀착.

### 5.4 단계별 로드맵 (현재 → 9월 말, verify-gate 방식)

| 단계 | 기간(체크포인트) | 핵심 작업 | **검증 게이트(완료 정의)** |
|---|---|---|---|
| **P0. 기반 정합** | 즉시(Day-0) | ① 라이센스 **실가용성** 확인(아래 모순), ② venv(Windows) 재생성, ③ **두 레포 1케이스 왕복 스모크**(통합리스크 #0), ④ 사내 LLM 어댑터 연결 PoC | Abaqus/OS/HM 1-run 성공 로그 + agent_cae3↔hm seam 왕복 1회 + H-Chat Pro 응답 1회 |
| **P1. 동어반복 해소 + 컴플라이언스** | ~7월 중순(센터레벨) | ① 실솔버 actual을 **독립 closed-form**으로 채점(solver≠validator), ② **inject_error FAIL 검출** 증명, ③ `internal_adapter` 사내 LLM **실구현**, ④ KPI 측정셋 설계 | 1개 선형 케이스: 정상→PASS·주입오류→**FAIL 검출 1건 이상** + 런타임 LLM이 사내 |
| **P2. 통합·MCP·1차 KPI** | ~8월 초(연구소레벨) | ① **MCP 어댑터(typed tools)** 1차, ② VeriCAE-Pilot **seam 실연결**(agent_cae3 상위 + hm 하위), ③ 성공률·개입횟수·**오결과 차단율** 1차 측정 | typed tool 5개 외부 클라이언트 호출 성공 + 통합 1케이스 E2E + KPI 표 1판 |

> **라이센스 모순 해소(P0 0순위)**: durable 기록(2026-06-14)은 사용자가 **Abaqus·HyperMesh 2024·OptiStruct·nCode DesignLife 보유**라 하나, hm-assistant 코드 실측은 "실솔버 실행 이력 0·가용성 미확인"이다. → **보유는 맞되 레포가 아직 한 번도 실행하지 않은 상태**로 해석. P0 스모크로 실가용성을 코드로 확정해야 P1~P3 전제가 성립한다.

### 5.5 정량 검증 지표(KPI) 체계 — 측정 프로토콜 (부록 C 상세)

원본·코드·경쟁자 어디에도 없는 **가장 큰 공백**. 9월 실증의 핵심은 이 측정이다.

- **무엇을(테스트셋)**: 부품유형별 N개(예: 선형정적 브라켓 5 + 모달 2 + 피로 2), 문제유형 라벨 부착.
- **무엇 대비(baseline)**: 동일 과제의 **수작업 소요시간/품질** + LLM 1-shot(게이트 없음).
- **지표 정의**: ① 성공률(E2E 완주), ② 평균 **개입 횟수**(사람 수정), ③ **오결과 차단율**(주입오류 FAIL 검출/총 주입), ④ 게이트 **위양성률**(정상을 FAIL로), ⑤ 단계별 before/after 소요시간.

### 5.6 사내도구 정합성 & 공모전 적합성

**(a) 컴플라이언스(필수)** 〔실측〕: 런타임 LLM이 사외 Gemini인 현 상태는 '사내 도구 only' 규정 저촉. → **사내 LLM(H-Chat Pro) 어댑터 실구현이 P1 필수**. 에이전트 표면은 경쟁자처럼 **Claude Code(허용 도구)** 활용을 권장(컴플라이언스 + 생산성). **LangGraph는 오케스트레이션 라이브러리**(AI 어시스턴트/서비스가 아님)라 규정 대상이 아닐 가능성이 높으나, **주최측 확인을 권고**(열린 쟁점).

**(b) register 번역**: 내부 전략문서의 '해자/시장/사업화' 언어를, 공모전 제출 시 **'개인 업무개선 PoC/활용사례'** 언어로 번역한다(공모전은 개인 PoC 수준, 서비스화 배포본 제외). 좁은 업무 1개로 앵커링(예: 선형 브라켓 해석 셋업→솔브→이론값 검증→보고서를 자연어로).

**(c) 심사 4축 매핑**:

| 심사축(배점) | 우리 근거 |
|---|---|
| 창의혁신성(30) | '라이브 제어'가 아니라 **생성자 독립 이론값 oracle을 문제유형별 레이어로 분리 + 4중 교집합 + nCode 미답영역** |
| 실현/확대가능성(30) | **데이터·보안 선행조건(사내 LLM 필수·CAE IP) + 직군 민주화 확산 + 단계적 일정**. (단일벤더 종속은 정직히 리스크로) |
| 효과성(20) | **시나리오 밴드 + 측정 프로토콜 약속**(자사 baseline 9월 확보) |
| AI과제적합성(20) | **예측형 AI(ML/DL) 경계선 명시** + 사내도구 only + LLM 오케스트레이션·검증 가드레일 |

**(d) 트랙 선택 권고**: hm-assistant(Mock)를 '활용사례'로 내면 미실증이 노출된다. **REAL Abaqus·실 .odb·독립 게이트를 가진 agent_cae3를 활용사례 실증 본체**로 내세우고, hm-assistant는 'HyperMesh 라이브 제어 프런트'로 배치하는 **VeriCAE-Pilot** 구도가 현실적(통합리스크 #0·epistemic 비독립은 각주화).

## §6. 결론

1. **종합·검증 결과**: 전략 원본 3종의 큰 방향(검증 중심·자동차 도메인)은 옳으나, **"RPC 유일 해자·이론값 만능 가드레일·E2E 차별·정량 우위"라는 강한 서사는 외부 선례(times1234·Abaqus-Control-MCP·PyMAPDL-MCP·FEABench)와 코드 실측(동어반복·stub·사외 LLM), 그리고 경쟁자 예시1과의 정면 비교 앞에서 과장**으로 판명된다. GPT 요약은 원본에 충실했고, GPT 비평의 핵심("RPC≠해자, 하이브리드가 정답")은 외부 1차출처로 **확정**된다.
2. **가치 예측**: 효율(후처리·리드타임·재작업)은 **방향성은 외부 프록시로 지지되나 자사 정량근거는 부재** — 가설로 두고 9월 KPI로 입증한다. 진짜 **새로운 가치는 "검증 레이어"**(오결과 차단)와 **민주화·지식자산화·추적성**이며, 예측형 AI(ML surrogate)는 경계 밖이다.
3. **현 시점 유효 계획**: **내부 세션엔진 + 외부 MCP 어댑터 + 사내 LLM + typed tools + 다층 검증게이트**(Ansys 공식 패턴과 동형)로 수렴하되, **동어반복 해소·사내 LLM 컴플라이언스·정량 KPI** 세 가지를 P0~P3로 실증하는 것이 핵심이다. 해자는 "선언"이 아니라 **"오결과 차단율 같은 숫자"**로 입증된다.

**한 줄 결론**: *Hyper-Pilot의 미래는 "상용 GUI를 직접 제어한다"가 아니라(그건 이미 흔하다), "**생성자와 독립된 물리 검증으로 틀린 해석을 걸러낸다**"를, 자동차 내구(nCode) 도메인에서, 사내 도구로, 숫자로 입증하는 데 있다.*

### 부록 F. 용어집

- **동어반복(self-refuting/tautology) 검증**: 검증자가 솔버와 같은 식·입력을 써 무조건 PASS가 되는 무의미 검증. 본 프로젝트 hm-assistant의 핵심 약점.
- **code-vs-code 독립 vs epistemic 독립**: 전자=구현이 분리됨(좌표·단위·BC 오류 검출), 후자=입력출처까지 분리됨(입력추출·공식 체계오차 검출). agent_cae3는 전자만 달성.
- **closed-form 이론값 oracle**: 빔/판/브라켓 등 선형 문제의 해석해(δ=FL³/3EI 등)로 솔버 결과를 채점하는 독립 기준.
- **MCP**: Model Context Protocol. 데이터계층이 JSON-RPC 2.0인 AI 에이전트용 표준 도구계층.
- **the seam(통합 접합점)**: VeriCAE-Pilot에서 agent_cae3(상위)↔hm-assistant(하위)를 잇는 단일 인터페이스 `run_full_analysis(...)→(bool, 8키 dict)`.
- **4중 교집합**: HyperWorks 생태계 × 라이브 세션 × closed-form 게이트 × 자동차 부품.
